## Task 1 — 검증 스크립트 구축

- **상태:** 완료
- **산출:** `tools/validate_problems.py`, `tools/samples/valid_sample.md`, `tools/samples/invalid_sample.md`, `tools/samples/malformed_sample.md`, `tools/samples/silent_failure_sample.md`, `tools/samples/heading_edgecase_sample.md`
- **수치:** 정상 픽스처 3항목 PASS / 위반 픽스처 HARD 3건 검출. 코드 리뷰 3회를 거치며 픽스처를 5개로 확장하고, 조용한 실패 경로 4건을 수정: (1) 어긋난 ID 헤딩이 오류 없이 조용히 파싱에서 빠지던 것, (2) 어긋난 헤딩 검출 후 `current`가 리셋되지 않아 직전 정상 블록의 필드가 오염되던 것, (3) 닫히지 않은 코드펜스가 파일 끝까지를 "펜스 안"으로 간주해 이후 문제 블록 전체를 조용히 무시하던 것, (4) 한 블록 안에서 같은 필드를 두 번 쓰면 나중 값이 이전 값을 조용히 덮어쓰고 PASS 처리되던 것.
- **판단:** 발산보다 검증기를 먼저 만들어, 에이전트가 스스로 형식을 교정하게 함. 초기 구현에는 "오류 없이 조용히 넘어가는" 경로가 여러 개 남아 있었고, 이런 경로는 발산 에이전트 6개가 실제로 형식을 어겼을 때 검증기가 이를 놓치고 PASS를 내주는 결과로 이어지므로, 코드 리뷰를 반복해 조용한 실패를 능동적으로 찾아 막는 방식으로 진행함.
- **다음 Task에 넘기는 것:** 근거 URL은 쉼표 구분 다중 값을 허용함. 코드펜스(```) 안의 내용은 파싱에서 완전히 제외되므로 형식 예시는 코드펜스로 감싸면 안전함. 단, 코드펜스는 반드시 닫아야 하며 닫히지 않으면 HARD 오류로 검출됨. ID 헤딩은 `### P{1-6}-{2자리}` 형식을 엄격히 요구하며, 자리수 부족(`### P1-1`)이나 부가 텍스트(`### P1-01: 제목`)는 HARD 오류로 검출됨. 한 블록 안에 같은 필드를 두 번 쓰면 HARD 오류로 검출됨.

## Task 2 — 리서치 헌장 작성

- **상태:** 완료
- **산출:** `docs/research/00-research-brief.md`, `docs/research/prompts/divergent-template.md`
- **수치:** 헌장 101줄 (상한 140줄)
- **판단:** 브리프의 헌장 본문과 프롬프트 템플릿 내용은 그대로 옮겨 적었고, Task 1에서 추가로 확정된 검증기 동작(어긋난 ID 헤딩 검출, 블록 내 필드 중복 검출) 두 줄만 §8에 덧붙였다. 헌장을 실제 검증기(`tools/validate_problems.py`)로 직접 돌려 `검사 대상: 0개 항목` / `PASS`를 확인함으로써, 헌장 안의 형식 예시(§4)와 요약 블록 규약(§8)이 검증기에 의해 실제 문제 항목으로 오인되지 않음을 실증했다. 이는 문서만 읽고 "형식이 맞겠지" 추정하는 대신, 다음 단계에서 6개 발산 에이전트가 동시에 이 헌장을 프롬프트에 삽입해 쓸 때 자기 파일이 아니라 헌장 자체 때문에 검증이 오염될 위험을 사전에 차단하기 위함이다. 이후 리뷰에서 Critical 1건·Important 4건·Minor 3건이 지적되어 모두 반영했다: (1) 템플릿이 §8 요약 블록 작성 지시를 실행 지시("반드시 지킬 것"·"산출 방법")에서 빠뜨려 검증기가 잡지 못하는 누락 위험이 있었던 것을 세 곳(반드시 지킬 것 1번, 산출 방법 전체, 자체 검증 말미)에 명시적으로 추가해 막음, (2) §4 `당사자` 금지 예시가 "소상공인" 한 단어뿐이라 검증기 `VAGUE_ACTORS` 11개 목록과 불일치했던 것을 목록 전체로 맞춤, (3) §7 공고 예시 7개의 해결책 오염 위험이 P1 축 서술과 §7 경고문 양쪽에서 약했던 것을 양쪽 다 강화, (4) §5 규칙 4(개수 채우기)가 근거 품질을 밀어낼 유인을 만들던 것을 "근거 없는 항목으로 채우지 말 것" 규칙으로 보완, (5) 이 문장 자체의 자기모순(항목 3개를 나열하며 "두 줄"이라 칭했고, 실제로는 §8에 없는 "코드펜스 닫힘 필수"를 있다고 잘못 기록한 것)을 바로잡음. Minor 3건(템플릿 플레이스홀더 개수 5→6 정정 및 `{{RESEARCH_BRIEF}}` 목록 포함, §4 ID 형식에 축번호 범위 명시, §4 "필드 순서를 바꾸지 말 것"을 자동 검증 대상이 아님을 밝히는 문구로 완화)도 함께 반영했다.
- **다음 Task에 넘기는 것:** 헌장(`00-research-brief.md`)과 템플릿(`divergent-template.md`) 모두 검증기로 실측 검증되어 `검사 대상: 0개 항목` / `PASS` / `exit=0`임을 확인함 — 두 문서를 발산 에이전트 프롬프트에 그대로 삽입해도 검증기가 문서 내 예시를 실제 문제 항목으로 오인하지 않는다. 다음 Task(발산 에이전트 실행/오케스트레이션)에서는 `divergent-template.md`의 플레이스홀더 6개(`{{AXIS_ID}}` `{{AXIS_NAME}}` `{{AXIS_SCOPE}}` `{{MIN_COUNT}}` `{{OUTPUT_PATH}}` `{{RESEARCH_BRIEF}}` — 앞 5개는 축별 값, `{{RESEARCH_BRIEF}}`는 전 축 공통 헌장 전문)를 채워 6개 축(P1~P6, §6 표 기준 최소 산출 개수 상이)에 대해 서브에이전트를 병렬 실행하면 된다. 각 에이전트는 산출 후 `python tools/validate_problems.py {산출파일} --min {최소개수}`로 자체 검증하도록 템플릿에 이미 지시되어 있다. 여기에 더해 `--check-urls`를 **권고 절차(게이트 아님)**로 추가했다 — 형식 검증은 URL 실재를 보지 않으므로 환각 URL이 PASS를 받을 수 있으나, 공공기관 사이트의 봇 차단(403)으로 인한 오탐도 흔하므로 자동 탈락시키지 않고 수동 확인하도록 지시한다.

## Task 3 — 발산 축 P1~P3

- **상태:** 완료
- **산출:** `docs/research/01-divergent/p1-notice-reverse.md`, `p2-industry.md`, `p3-persona.md`
- **수치:** P1 20개 / P2 21개 / P3 21개 (합계 62개), 스키마 검증 3파일 전부 PASS(exit=0), 요약 블록 위치 전부 정상(3행 < 첫 문제 블록 10행)
  - P2 업권 분포: 은행 5 / 인터넷은행 4 / 증권 4 / 생보 4 / 보안 4
  - P3 페르소나 분포: 청년 5 / 고령층 3 / 장애인 4 / 외국인 3 / 창업자 3 / 자영업자 3
- **판단:** WARN 1건(P2-18 '에이전트')은 삭제하지 않고 유지했다. 해당 항목에서 'AI 에이전트'는 제안하는 해결책이 아니라 **공격받는 대상**(금융회사가 도입한 AI 에이전트가 고도화 공격에 노출됨)을 가리키는 명사다. 검증기의 `SOLUTION_KEYWORDS`가 단순 부분문자열 매칭이라 발생한 예상된 오탐이며, 이 한계는 Task 1 리뷰에서 이미 Minor로 기록되어 있다. 당사자 뭉뚱그림 WARN은 3축 전부 0건 — 헌장 §4에 검증기 `VAGUE_ACTORS` 11개 목록을 그대로 명시한 조치가 효과를 냈다.
- **다음 Task에 넘기는 것:**
  - P4~P6 실행 시 프롬프트 수정 필요 사항 **없음**. 헌장·템플릿이 의도대로 작동했다(3축 모두 자체 검증을 통과한 상태로 반환).
  - 근거 강도가 약해 Task 6(실재성 검증)에서 우선 확인이 필요한 항목: **P1-06**(고령층 보이스피싱 피해비중 — 지역 매체 인용, 원출처는 2025년 9월 국회 행안위 공개 경찰청 자료), **P3-15**(모기지신용보험 차별 — 2010년 인권위 결정으로 통계가 오래됨), **P3-16·P3-17**(자금조달 통계 — 1차 출처 원문 대조 미완).
  - P3-10(kca.go.kr)은 SSL 인증서 체인 오류로 `--check-urls` 자동 판정에 실패했으나 curl로 HTTP 200 및 본문 일치를 확인했다. Task 6에서 같은 현상이 재발할 수 있으니 자동 실패를 곧바로 탈락으로 처리하지 말 것.
