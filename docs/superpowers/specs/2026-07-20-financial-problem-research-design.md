# 2026 금융 AI Challenge — 금융 현안 발굴 리서치 설계

- 작성일: 2026-07-20
- 상태: 승인됨
- 범위: **금융 현안·문제 발굴 리서치**에 한정. 해결책 설계와 주제 확정은 별도 사이클.
- 진행 방식: **5단계 게이트 방식** (일자 고정 없음)

## 1. 배경과 목적

### 1.1 대회 제약 (공고 분석 결과)

| 항목 | 내용 |
|---|---|
| 기획서 + MVP 마감 | 2026-09-07(월) 10:00 — 작성일 기준 **49일** |
| 제출물 | 기획서(PDF) + 기능명세서(PDF) + **실행 가능한 웹서비스 URL** |
| 서비스 가동 보장 | 2026-09-07 11:00 ~ 2026-09-11 23:59 (미가동 시 결격) |
| 데이터 | **미제공** — 확보는 전적으로 참가자 책임 |
| 팀 | 최대 4인 |
| 1차 심사 | 100% 내부 비공개 심사, 기준은 **"주제 적합성, 부적격 여부"** |
| 발표 심사 | 상위 11팀 내외, 2026-10-13 오프라인 (PT 15분 + QA 5분) |
| 주최/주관 | 금융보안원 (후원: 금융위원회) |
| 공동개최사 | 하나은행, 신한은행, 카카오뱅크, KB증권, 생명보험협회 |

### 1.2 공고에서 도출한 전략적 함의

1. **주최가 금융보안원이다.** 세부주제 예시 7개 중 3개가 보안·이상거래·AI보안 계열이다. 심사위원의 도메인 이해도가 보안 영역에 편중될 가능성이 높다.
2. **우수상 5개가 공동개최사 이름을 달고 있다.** 은행 3, 증권 1, 보험 1의 슬롯이 사실상 배분되어 있다. 특정 업권에 명확히 귀속되는 문제일수록 수상 경로가 뚜렷하다.
3. **1차 심사 기준이 "주제 적합성"이다.** 1차는 탈락시키는 심사이고, 실질 승부는 11팀 안에서의 발표 심사다.
4. **"데이터 미제공"이 최대 리스크다.** 대부분의 팀이 더미 데이터 데모로 끝난다. 실데이터 확보 경로 확보 자체가 차별점이 된다.

### 1.3 팀 조건

- 참가 형태: **3~4인 팀**
- 강점: ML/데이터 분석, LLM/Agent 개발, 금융 도메인 지식
- **공백: 웹 풀스택 / 배포** — 본 리서치 범위 밖이나, 해결책 설계 사이클에서 최우선 제약으로 반영해야 함
- 가용 시간: **불확정** — 따라서 일자 고정 계획 대신 단계별 완료 게이트로 통제한다

### 1.4 진행 통제 원칙

일자를 고정하지 않으므로, 각 단계는 **완료 게이트**로 종료를 판정한다. 다만 게이트만 두면 완벽주의로 단계가 늘어지므로, 모든 단계에 **최소 산출량(하한)과 중단 조건(상한)을 함께** 정의한다.

- **완료 게이트** — 이 조건을 채우면 다음 단계로 넘어간다
- **중단 조건** — 이 선을 넘으면 미완이어도 강제로 다음 단계로 넘어간다

작업량 기준 총 소요는 5~7일 상당으로 추정한다. 이보다 크게 초과하면 리서치가 아니라 회피가 진행 중인 것이므로, 중단 조건을 적용한다.

## 2. 리서치 범위 정의

### 2.1 이번 사이클이 산출하는 것

**검증된 금융 현안 목록과 우선순위.** "주제"나 "서비스 아이디어"가 아니다.

```
[이번 리서치]  금융 현안·문제 발굴  ──→  유망 문제 영역 3~5개 + 문제 정의서
                                              │
                                              ↓  (별도 사이클)
[다음 단계]                            해결책 설계 → 주제 확정 → MVP 개발
```

### 2.2 문제 중심으로 설계한 근거

문제 발굴과 해결책 설계를 한 단계에서 섞으면, "지금 AI로 풀 방법이 떠오르는 문제"만 후보에 남는다. 탐색 범위가 조용히 좁아지고, 미개척 영역일수록 먼저 탈락한다. 따라서 발산 단계에서 해결책 관련 서술을 **금지 사항으로 명시**한다.

대회 공고의 "세부주제 예시" 7개도 이미 해결책 형태로 서술되어 있다("AI 맞춤형 금융 매칭 서비스" 등). 이를 출발점으로 삼으면 타인의 해결책 프레임 안에 갇히므로, **예시가 전제하는 근본 문제를 역추출**하는 방식으로 사용한다.

### 2.3 접근법 선택

세 가지 접근을 검토했다.

| 안 | 성격 | 채택 여부 |
|---|---|---|
| A. 축 전수 스윕 (Breadth-first) | 문제 공간을 축으로 쪼개 병렬 발산 | **채택 — 발산 단계 뼈대** |
| B. 선행·동향 역산 (Gap-first) | 기존 대응 지도화 후 공백 탐색 | **채택 — 검증 단계에 흡수** |
| C. 데이터 실재성 역산 (Data-first) | 확보 가능 데이터에서 역으로 도출 | **보류 — 다음 사이클로 이연** |

**C안을 발산 단계에 두지 않은 이유:** 데이터를 먼저 조사하면 "DART가 있으니 기업분석을 하자" 식으로 데이터에 끌려다니는 발상이 나온다. 발상을 먼저 펼치고 실현 가능성을 나중에 따지는 순서가 결과 품질이 높다. 다만 완전히 배제하면 다음 사이클이 맨땅에서 시작하므로, Phase 5 산출물에 **점수 미반영 참고 메모**로만 첨부한다(§7 리스크 3).

## 4. 문제 스키마

모든 문제 항목은 아래 고정 스키마를 따른다. 병렬 에이전트 6개가 각자 다른 형식으로 산출하면 통합 비용이 발산 비용을 넘어서므로, 스키마 고정이 병렬화의 전제 조건이다.

| 필드 | 규칙 |
|---|---|
| `ID` | `P{축번호}-{일련번호}` (예: `P1-01`) |
| `문제 한줄` | **"[누가] [어떤 상황에서] [무엇 때문에] [어떤 손해·불편을 겪는다]"** 템플릿 준수. 벗어나면 반려 |
| `당사자` | 구체적 집단. "소상공인" 같은 뭉뚱그림 금지, "개업 1년 미만 요식업 자영업자" 수준으로 |
| `발생 맥락` | 언제·어디서 발생하는가 |
| `피해 규모` | 정량 수치 + 출처. 없으면 `수치없음` 명시 |
| `현재 대응` | 지금 어떻게 처리되고 있는가 (수작업 / 부분 해결 / 방치) |
| `근거 URL` | 최소 1개. 없으면 항목 폐기 |
| `관련 업권` | 은행 / 인터넷은행 / 증권 / 생보 / 보안 / 무관 |

### 4.1 발산 단계 금지 사항

프롬프트에 명시하여 강제한다.

- 해결책·서비스·기능을 서술하지 말 것
- "AI로 ~하면 된다" 류의 서술 금지
- 기술 스택·모델·기법 언급 금지
- 오직 **"무엇이 잘못되어 있는가"**만 기술할 것

`피해 규모`를 모를 때 `수치없음`으로 적도록 허용하는 것도 같은 취지다. 억지로 채우게 하면 존재하지 않는 통계를 생성한다.

### Phase 2 — 발산 (병렬 6축 문제 발굴)

| 축 | 조사 대상 | 최소 산출 |
|---|---|---|
| **P1** | 공고 세부주제 예시 7개가 전제하는 근본 문제 역추출 + 인접 문제 확장 | 15개 |
| **P2** | 업권별 현안 — 업계 리포트, 협회 자료, 금융지주 IR·연차보고서의 리스크 언급 | 15개 |
| **P3** | 소비자 페르소나별 페인포인트 — 금융민원 통계, 소비자원 상담 사례, 실태조사 | 15개 |
| **P4** | 채널·프로세스 마찰 — 창구/앱/콜센터/오픈뱅킹에서 사용자가 이탈·실패하는 지점 | 10개 |
| **P5** | 감독당국·정책이 공식 지목한 문제 — 금융위·금감원·금융보안원 보도자료, 국정과제, 제재·지적 사항 | 15개 |
| **P6** | 사고·피해·손실 통계 — 보이스피싱, 보험사기, 금융사기, 디지털 소외 실태 | 10개 |

목표: 원시 문제 80개 내외 → 중복 제거 후 **40~60개**

**축 P5의 근거:** 심사 주체가 금융보안원, 후원이 금융위다. 감독당국이 공식 문서에서 명시한 문제를 인용하면 문제 정의의 정당성을 두고 다툴 여지가 사라진다.

**축 중복은 의도적이다.** P2(업권)와 P3(페르소나)는 겹친다. 겹치는 지점에서 같은 문제가 반복 도출되면, 그 중복 빈도 자체가 문제 실재성의 신호가 된다(Phase 5에서 가점).

#### 발산 에이전트 공통 지침

1. **판단 금지** — 풀기 어렵다는 이유로 스스로 탈락시키지 말 것
2. **해결책 금지** — §4.1 준수
3. **정량 근거 우선** — 수치 있는 문제를 우선 수집, 없으면 `수치없음` 표기
4. **최소 개수 충족** — 부족하면 축을 더 잘게 쪼개서라도 채울 것
5. **모든 항목에 출처 URL 첨부**

"판단 금지" 지시를 생략하면 에이전트는 자기 검열을 하여 무난한 후보만 보고한다. 발산 단계에서 이는 치명적이다.

**완료 게이트**
- [ ] 6개 축 파일 모두 생성됨
- [ ] 각 축이 최소 산출량을 충족함
- [ ] 중복 제거 후 `02-problem-pool.md`에 **40개 이상** 남음
- [ ] 스키마 위반 항목(필드 누락·해결책 서술 포함)이 0건

**중단 조건** — 중복 제거 후 60개를 넘으면 추가 발산을 멈춘다. 검증 단계 비용이 개수에 비례하므로, 더 모으는 것보다 거르는 편이 낫다.

### Phase 3 — 검증 (문제 실재성 확인)

발산 산출물은 아직 "주장"이다. 이 단계에서 사실로 승격시키거나 폐기한다.

#### 3-1. 근거 등급 판정

| 등급 | 출처 유형 |
|---|---|
| **A** | 정부·감독기관 공식 통계·보도자료 (금융위, 금감원, 금융보안원, 통계청) |
| **B** | 협회·연구기관 보고서 (은행연합회, 생명보험협회, 보험연구원, 금융연구원) |
| **C** | 언론 보도 |
| **D** | 일화·블로그·근거 없는 추정 |

#### 3-2. 반증 탐색 (Adversarial Check)

각 문제에 대해 **"이 문제는 이미 해결되었다"는 증거를 적극적으로 탐색**한다. 검증 에이전트에게 지지를 요청하면 확증 편향이 그대로 반영되므로, 반증을 과제로 부여한다.

확인 항목:
- 이 문제를 해결하는 기존 상용 서비스가 이미 존재하는가?
- 이 문제를 제거하는 제도·규제 변화가 이미 시행되었거나 예정되어 있는가?
- 인용된 통계가 2023년 이전 데이터는 아닌가?

#### 3-3. 탈락 기준

| 조건 | 처리 |
|---|---|
| 근거 등급 D이고 상향 근거 확보 실패 | 탈락 |
| 반증 탐색에서 "이미 해결됨" 확인 | 탈락 |
| 근거 URL이 존재하지 않거나 내용 불일치 (환각) | 탈락 |
| 공고 주제 범위(금융) 밖 | 탈락 |
| 그 외 | 생존 → Phase 4 |

생존 예상: **20~30개**

**URL 실증은 생략 불가하다.** 발산 에이전트가 생성한 링크를 실제로 열어 내용 일치를 확인하지 않으면, 존재하지 않는 통계 위에 7주짜리 프로젝트를 쌓게 된다.

**완료 게이트**
- [ ] 전 항목에 근거 등급(A~D) 부여 완료
- [ ] 전 항목에 반증 탐색 결과 1줄 이상 기록
- [ ] 생존 항목의 근거 URL **100% 실제 접속 검증** 완료
- [ ] 생존 항목 **15개 이상**

**중단 조건** — 생존이 15개 미만이면 Phase 2로 되돌아가 축을 추가한다. 40개를 넘으면 근거 등급 A/B 항목만 남기고 잘라낸다.

---

### Phase 4 — 평가 (우선순위 스코어링)

해결 가능성은 평가하지 않는다. 문제 자체의 매력도만 본다.

| 축 | 질문 | 가중치 |
|---|---|---|
| ① 실재성 | 근거가 얼마나 단단한가 | ×1.0 |
| ② 심각도 | 당사자 1인이 겪는 고통·손실이 얼마나 큰가 | ×1.0 |
| ③ 규모 | 몇 명이 / 얼마의 금액이 영향받는가 | ×1.0 |
| ④ 미해결도 | 지금 방치되어 있거나 수작업으로 처리되는가 | **×1.5** |
| ⑤ 심사 적합성 | 공고 주제 범위·주최기관 관심사와 얼마나 맞는가 | **×1.5** |
| ⑥ 시의성 | "왜 지금인가"에 답할 수 있는가 | ×1.0 |

각 축 1~5점. 가중치 근거:
- **④ 미해결도** — 이미 잘 해결되는 문제를 다시 푸는 것은 완성도와 무관하게 평가받지 못한다.
- **⑤ 심사 적합성** — 공고에 1차 심사 기준이 "주제 적합성"으로 명시되어 있다. 여기서 걸리면 나머지 축이 무의미하다.

#### 4-1. 점수 앵커

앵커를 정의하지 않으면 20~30개 채점 과정에서 기준이 표류하여 순위가 노이즈가 된다.

**① 실재성**
- 5점 — A등급 근거 + 최근 2년 내 정량 수치
- 3점 — B~C등급 근거 + 수치 있음
- 1점 — 수치 없음, 정성 서술만

**④ 미해결도**
- 5점 — 현재 아무 대응이 없거나 전적으로 수작업
- 3점 — 부분적 해결책은 있으나 사각지대가 넓음
- 1점 — 이미 성숙한 상용 서비스가 존재

**⑤ 심사 적합성**
- 5점 — 공고 주제 범위 내 + 주최(금융보안원) 또는 공동개최사 업권에 명확히 귀속
- 3점 — 공고 주제 범위 내이나 특정 기관 귀속이 불분명
- 1점 — 금융 주제이나 공고 취지와 거리가 있음

**②③⑥** — 5점: 명확·큼·강함 / 3점: 보통 / 1점: 약함. 채점 시 근거 한 줄을 함께 기록한다.

**완료 게이트**
- [ ] 전 생존 항목에 6축 점수 + 채점 근거 1줄 기록
- [ ] 가중 합계 기준 정렬 완료
- [ ] 상위 8~12개 식별

**중단 조건** — 점수 재조정을 2회 이상 반복하지 않는다. 3회째 조정은 순위를 바꾸기 위한 사후 합리화일 가능성이 높다.

---

### Phase 5 — 수렴 (문제 영역 선정)

1. 상위 8~12개를 문제 성격별로 클러스터링 (예: 사기·보안 / 포용금융 / 기업금융 / 프로세스 효율)
2. **여러 축에서 중복 도출된 문제에 가점** — 독립적으로 반복 발견된 문제는 실재성이 높다
3. **유망 문제 영역 3~5개** 선정
4. 각 영역에 대해 **1페이지 문제 정의서** 작성
   - 문제 서술 / 당사자 / 피해 규모(수치) / 현재 대응 / 왜 지금인가 / 근거 출처
   - **해결책 항목은 비워 둔다**
   - `데이터 지형 메모`를 참고 정보로 첨부 (점수 미반영)
     - 작성 대상: 선정된 3~5개 영역에 한해서만
     - 작성 범위: 해당 문제 영역에 접근 가능한 공개 데이터·API가 존재하는지 여부와 명칭까지만. 활용 방안은 쓰지 않는다
     - 상한: 영역당 30분. 초과 시 `미확인`으로 남긴다
5. `05-shortlist.md` 작성 → 리서치 사이클 종료

**완료 게이트**
- [ ] 유망 문제 영역 3~5개 선정
- [ ] 각 영역에 1페이지 문제 정의서 작성 완료
- [ ] 각 문제 정의서에 해결책 서술이 없음을 확인
- [ ] 각 문제 정의서에 데이터 지형 메모 첨부 (`미확인` 허용)

**중단 조건** — 5개를 초과해 선정하지 않는다. 다음 사이클(해결책 설계)의 부하가 영역 수에 비례한다.

## 6. 전체 파이프라인

```
Phase 1  준비   리서치 헌장 확정 (스키마·금지사항·축 정의)
   ↓  게이트: 헌장에 스키마·금지사항 명문화
Phase 2  발산   6축 병렬 → 문제 80개 → 중복제거 40~60개
   ↓  게이트: 풀 40개 이상, 스키마 위반 0건
Phase 3  검증   근거등급 + 반증탐색 + URL실증 → 생존 15~30개
   ↓  게이트: URL 100% 실증, 생존 15개 이상
Phase 4  평가   6축 스코어링 (미해결도·적합성 가중) → 상위 8~12개
   ↓  게이트: 전 항목 채점 근거 기록
Phase 5  수렴   클러스터링 → 유망 문제영역 3~5개 + 문제정의서
   ↓
        [별도 사이클] 해결책 설계 → 주제 확정 → MVP 개발
```

작업량 추정: 총 5~7일 상당. 실제 기간은 가용 시간에 따라 가변한다.

## 7. 리스크와 대응

| # | 리스크 | 대응 |
|---|---|---|
| 1 | **근거 URL 환각** — 존재하지 않는 통계·링크 생성 | Phase 3에서 전 URL 실제 접속·내용 대조 |
| 2 | **팀의 웹 배포 역량 공백** | 본 리서치 범위 밖. 해결책 설계 사이클에서 최우선 제약으로 반영 |
| 3 | **해결 가능성 미검토로 인한 재작업** | Phase 5 shortlist에 `데이터 지형 메모`를 참고 정보로만 첨부. 점수 미반영 |
| 4 | **단계 지연 / 무한 확장** | 각 단계에 완료 게이트와 중단 조건을 병행 적용 (§1.4) |
| 5 | **한국 금융 자료 접근 한계** | 감독기관 보도자료·협회 통계를 1순위 경로로 고정 (P5/P6 축) |
| 6 | **발산 축 간 결과 형식 불일치** | §4 스키마를 발산 프롬프트에 전문 삽입하여 강제 |
| 7 | **마감 압박에 의한 단계 생략** | Phase 3(검증)은 생략 불가로 지정. 나머지 단계는 산출량을 줄이되 절차는 유지 |

**리스크 3에 대한 판단:** 문제만 보고 선정한 3~5개가 전부 데이터 확보 불가능한 영역일 가능성이 있다. 이를 스코어링에 반영하면 §2.2에서 배제한 편향이 되돌아오므로, 점수 대신 메모로 남기는 절충을 택한다. 다음 사이클의 출발점 역할만 하게 한다.

**리스크 7에 대한 판단:** 시간이 부족해지면 가장 먼저 생략하고 싶어지는 것이 검증 단계다. 그러나 검증을 건너뛴 문제 위에 세운 기획서는 발표 심사 질의응답에서 무너진다. 발산 산출량을 절반으로 줄이는 편이 검증을 생략하는 것보다 낫다.
